## Mission
Build the **batch enumerator** that grids parameters per `docs/exhaustion.txt`, runs each case through the kernel via the existing loader, and writes one long-format result CSV plus one metadata JSON. The enumerator must:

- generate `CaseRecord` values directly in memory from a Python `GridSpec` constant вҖ” no intermediate CSV file, no DSL parser
- apply analytical domain reduction when prerequisites hold (`a1 = a2 = 0.5` AND both war-payoff toggles enforced); fall back to the full grid otherwise вҖ” both paths gated by an explicit predicate, never both running
- reuse `loader.runner.run` and `loader.writer.flatten` as-is вҖ” no edits to the loader, no duplication of column logic
- write exactly two files: a long-format `cases.csv` (loader's existing column set) and a `metadata.json` (spec snapshot, options, case count, runtime, reduction-path flag, truncation flag)
- ship as a top-level CLI in parallel with the loader CLI вҖ” `python -m gaming_research.exhaustion -o cases.csv [...]`

The kernel and loader are both **locked**. The exhaustion package is the only thing built this phase.

### Architecture (locked by the design review)
- **Two parallel top-level CLIs**: `gaming_research.exhaustion` (research entry) and `gaming_research.loader` (ad-hoc CSV entry). No unified subcommand wrapper.
- **Dependency direction**: `exhaustion вҶ’ loader.runner / loader.writer вҶ’ kernel.evaluate_case`. Exhaustion never calls kernel directly.
- **Spec source**: `GridSpec` frozen dataclass and `CURRENT_SPEC` constant live in `exhaustion/spec.py`. `docs/exhaustion.txt` stays as a doc-only reference; it is not parsed at runtime.
- **Decimal text**: every kernel-field value generated by the enumerator is the canonical `str(Decimal_value)` form. The loader's byte-for-byte passthrough then carries that exact string into `cases.csv`. Per-field formatting rules pin in the plan doc.
- **`case_id`**: parameter-encoded (`m1=2_m2=1_p=0.3_c1=0.7_c2=0.6`), generated by the enumerator; loader never auto-generates IDs in this path.
- **Safety gate**: estimated case count above the threshold pinned in the plan doc causes the CLI to exit non-zero unless `--allow-large-grid` is passed.

## Critical Rules
Non-negotiable. These override any local-feeling tradeoff.

- **Kernel is locked.** Do not edit anything under `src/gaming_research/kernel/`. Do not add fields to `Options`, `Params`, `KernelResult`, or any other kernel type.
- **Loader is locked.** Do not edit anything under `src/gaming_research/loader/`. Do not change `OUTPUT_COLUMNS`, `flatten()`, `run()`, `CaseRecord`, or `LoaderError`. The exhaustion package consumes the loader API as-is. If something is missing, stop and ask вҖ” do not "extend" the loader.
- **Stdlib only.** No pandas, no polars, no PyYAML, no toml-parser. Phase 3 adds zero new runtime dependencies. `pyproject.toml` deps stay at `["scipy"]` (and `pytest` in the dev extra).
- **No DSL parser.** `docs/exhaustion.txt` is a human-readable doc; do not parse it at runtime. The grid is encoded as a `GridSpec` Python constant in `spec.py`. Changing the grid means editing that constant.
- **In-memory pipeline only.** No intermediate parameter CSV is written between the enumerator and the runner. `enumerate_cases` yields `CaseRecord` objects directly into `loader.runner.run`. The only files written are `cases.csv` and `metadata.json`.
- **Decimal text is canonical and generated.** Every kernel-field value the enumerator produces uses `str(Decimal_value)` after `Decimal` arithmetic on the spec's step constants. The plan doc pins the per-field expected form (e.g. `min1 вҶ’ "2"`, `p вҶ’ "0.3"`, `c1 вҶ’ "0.7"`). The loader's byte-for-byte passthrough must then carry that exact string into `cases.csv` вҖ” the end-to-end golden file verifies this.
- **Reduction is gated by an explicit predicate.** `enumerate.py` checks `options.enforce_war_payoff_s1 and options.enforce_war_payoff_s2 and spec.a1 == Decimal('0.5') and spec.a2 == Decimal('0.5')`. Only when all four hold may the analytical narrowing of `c1` and `c2` per `(min1, min2, p)` apply. The chosen path is recorded in `metadata.json` as a string flag.
- **Two output files, user-specified path.** No `results/runs/<timestamp>/` directories, no auto-discovery. The CLI takes `-o cases.csv`; the metadata path is derived (the plan doc pins the exact derivation rule). No third file (no `summary.csv`, no `summary.md`).
- **I/O is confined.** Only `cli.py` and `writer.py` touch the filesystem or stdio. `spec.py`, `enumerate.py`, and `runner.py` must be pure: no `open`, no `print`, no `sys.stdout`, no `sys.stderr`. The CLI is the only place `print` / stderr writes are allowed.
- **Frozen dataclasses everywhere.** `@dataclass(frozen=True)` for `GridSpec` and any other new types. Use `tuple[...]` for sequence fields.
- **Single-threaded, single-process.** No multiprocessing, no threading, no asyncio. The 3920-case happy path is fast enough; the 14M-case fallback is gated behind `--allow-large-grid` and the user accepts the wall time.
- **Safety gate is mandatory.** When the estimated case count exceeds the plan-doc threshold, the CLI exits non-zero with a stderr message before any case runs. `--allow-large-grid` is the only override. The threshold value lives in the plan doc, not in code as an unnamed magic number.
- **CLI flags are minimal.** Phase 3 ships exactly the flags pinned in the plan doc вҖ” at minimum the loader's three Options pass-throughs (`--bluffing-mode`, `--enforce-war-payoff-s1`, `--enforce-war-payoff-s2`), `-o`, and `--allow-large-grid`. Do not add `--spec-file`, `--xtol`, `--workers`, or any other knob unless the plan doc explicitly lists it.
- **English messages only.** Stable codes (e.g. reduction-path flag values, exit-code semantics) are the contract; any human-readable text is short English.
- **No comments by default.** One-line *why* comments only. No multi-paragraph docstrings.
- **No new docs.** The only doc added in this phase is `exhaustion-implementation-plan.md` itself, written in step 1 below before any code. No edits to other existing docs.
- **README only gets the post-phase integration-guide edit.** No prose updates while implementing.

## Coding Order
Implement in this order. Each step should leave a runnable, importable state. Use `TodoWrite` to track these and commit after each meaningful step (commit message style: `MMDD-short-slug`, see `git log`).

1. `docs/exhaustion-implementation-plan.md` вҖ” write the *how* doc first, before any Python code in this phase. The user reviews and signs off before implementation begins. Must pin: module layout; `GridSpec` field list; `CURRENT_SPEC` values; per-field Decimal-text formatting table; reduction predicate exact text; `case_id` template; `metadata.json` field set; safety threshold default; CLI flag list and exit-code semantics; stderr summary text; fixture spec for the end-to-end test; golden file rows.
2. `src/gaming_research/exhaustion/__init__.py` and `src/gaming_research/exhaustion/spec.py` вҖ” `GridSpec` frozen dataclass; `CURRENT_SPEC` constant matching `docs/exhaustion.txt`; `estimate_case_count(spec, options)` for the safety gate; `reduction_eligible(spec, options) -> bool`. No I/O.
3. `src/gaming_research/exhaustion/enumerate.py` вҖ” `enumerate_cases(spec, options) -> Iterator[CaseRecord]`. Two paths: reduction (`c1 вҲҲ (pВ·max1, pВ·max1+a1)`, `c2 вҲҲ ((1-p)В·max2, (1-p)В·max2+a2)`, gated by `reduction_eligible`) and full grid (Decimal step iteration). `case_id` formatter and per-field Decimal-to-text formatter live here. No I/O.
4. `src/gaming_research/exhaustion/runner.py` вҖ” thin: iterates `enumerate_cases`, forwards each `CaseRecord` to `loader.runner.run`, yields `(record, result)` pairs. No I/O.
5. `src/gaming_research/exhaustion/writer.py` вҖ” `write_cases(pairs, cases_path)` calls `loader.writer.flatten` and writes `cases.csv` (LF, UTF-8, loader column order). `write_metadata(metadata, metadata_path)` writes `metadata.json` (sorted keys, UTF-8). Owns no column logic of its own.
6. `src/gaming_research/exhaustion/cli.py` + `__main__.py` вҖ” `python -m gaming_research.exhaustion -o cases.csv [...]`. Single argparse; safety-gate check before running; single loop; stderr summary; exit codes per the plan doc.
7. `tests/test_exhaustion_spec.py`, `tests/test_exhaustion_enumerate.py`, `tests/test_exhaustion_writer.py`, `tests/test_exhaustion_end_to_end.py` вҖ” fixtures live under `tests/fixtures/exhaustion/`. The end-to-end test uses a small spec (~10 cases) and compares both output files byte-for-byte against checked-in goldens.
8. Run `pytest` from the repo root; iterate until clean. Then run the CLI manually on the end-to-end fixture and confirm stderr summary and exit code match the plan doc. Also exercise the safety gate (with and without `--allow-large-grid`) on `CURRENT_SPEC` and confirm the gate fires.

## Anti-Features (Do Not Build)
These look helpful but are explicitly off-limits this phase:

- a parser for `docs/exhaustion.txt` (the grid lives in `spec.py` as a Python constant)
- a `--spec-file` / `--config` flag, YAML / TOML / JSON spec input
- pandas, polars, or any DataFrame-backed enumerator / writer
- a `summary.csv` or `summary.md` (grouped statistics are deferred; Phase 3 ships only `cases.csv` + `metadata.json`)
- a `results/exhaustion/<timestamp>/` directory tree or any auto-named output paths
- an intermediate parameter CSV between the enumerator and the loader runner
- parallel execution, multiprocessing, asyncio, or chunked processing
- resume / checkpoint / partial-output recovery
- per-case `Options` overrides driven by spec columns (Options is global to a run)
- a unified `python -m gaming_research <subcommand>` CLI (the parallel-CLI architecture is locked)
- any edit to `src/gaming_research/kernel/` or `src/gaming_research/loader/`, including "harmless" additions of helpers, re-exports, or new fields
- any new field on `Options`, `Params`, `KernelResult`, `Solution`, `Summary`, `Branch`, `Validation`, `Derived`, `CaseRecord`, or `LoaderError`
- a "convenience" `enumerate_to_csv(...)` facade in the kernel or loader package
- compatibility wrappers for `audit.txt` / `audit_display.json`
- emoji in code, comments, fixtures, or commit messages
- multi-paragraph docstrings or "what" comments

## When in Doubt
- If `exhaustion-design.md` and `exhaustion-implementation-plan.md` disagree on a **locked decision** (the three at the bottom of the design doc): the **design doc wins**. Stop and ask before implementing the disagreeing piece.
- If they disagree on a **non-locked recommendation** (e.g. the design doc's "жҺЁиҚҗиҫ“еҮә" suggests `summary.csv` / `summary.md`; the plan doc says two files only): the **plan doc wins**, because Phase 3 scope was narrowed during design review. The narrower scope is intentional.
- If something is genuinely missing from both docs: stop and ask. Do not invent. The per-field Decimal-text formatting table, the `case_id` template, the reduction predicate, the `metadata.json` field set, the safety threshold default, the stderr summary text, and the exit code semantics all must be pinned in the plan doc вҖ” if you cannot find them there, the plan doc is incomplete.
- If a test needs a fixture spec or golden file the plan doc's fixture section does not pin, stop and ask. End-to-end goldens cannot be authored ad-hoc; they go in the plan doc and get reviewed.
